# Refactoring: Facade → UseCase Separation
## 1. 문제상황
현재 서비스 구조에서 facade가 다음과 같은 역할을 수행함
- 비즈니스 로직처리
- 응답 dto생성
- 여러 service 조합
- 트랙젝션 관리

프로젝트를 진행할 수록 facade가 비대해지면서 God Class 형태가 되었고
변경시 영향 범위가 커지는 문제가 발생했다.

### 문제점
- 테스트 어려움
- 기능 추가 시 facade 수정 필요함 이때 불필요한 에러가 발생할 수 있다.
- 단일 책임 원칙 위반
- 의존성 증가
- 코드 가독성 저하
## 2. 원인분석
facade가 도메인 단위로 설계되었고, 그 안에서 유스케이스 단위 분리 없이 메서드가 추가되었기 때문에

## 3. 목표
facade단 함수 내 복잡한 로직은 usecase로 옮겨서
각 클래스가 하나의 책임만 받도록 한다. -> 단일 책임 원칙


## 4. 고려한 결정들
facade를 제거하고 usecase추가 하지만 controller단에서 불필요한 의존성이 증가함
facade를 유지하고 usecase 추가 -> 레이어가 하나 더 늘어난다.

## 5. 최종 결정
Facade 유지 + 각 비즈니스 로직 마다 usecase 추가

## 6. 변경 후 이점
- srp를 충족 시킬 수 있다. -> 각 use case마다 하나의 로직을 부여
- dip 충족 -> Facade가 30개의 의존성을 모두 알던 것에서 usecase가 해당 시나리오에 필요한 의존성만 갖게 되어 의존 범위가 최소화 된다.
- ocp -> 새 기능 추가시 새 usecase 클래스를 추가하는 것으로 확장 가능하다.
- 테스트 용이 -> 각 UseCase가 필요한 의존성만 가지므로 Mock 개수가 줄어 단위테스트 작성이 쉬워진다.

## 7. trade-offs
- 파일 수가 증가
- 레이어 증가로 인해 구조가 복잡해짐


트랜잭션 경계 = 하나의 비즈니스 시나리오가 성공/실패하는 단위
